01

Post-screen interface

Plumb turns a codebase into a semantic field.

Agents should not scrape a repository like a terminal transcript. They should ask for intent: where a symbol is defined, who calls it, what a type means, what changed, and whether an edit is still valid. Plumb answers through structured MCP tools backed by LSP.

Meaning

Code intelligence at the level engineers use.

Symbols, hovers, definitions, references, call hierarchy, type hierarchy, and diagnostics come from the language server, not text guessing.

Control

Mutation is explicit, bounded, and observable.

Writes use per-path locks, atomic rename, dirty-state checks, optional mtime fencing, rate limits, and LSP file-change notifications.

Continuity

The daemon keeps context alive between chats.

A shared daemon hosts warm workspace language servers, per-connection sessions, project config, stats, and workspace memory.

Safety model

Autonomy needs engineering constraints, not vibes.

  • Read tracker

    Strict mode can require the file to be read in the current session before an edit is accepted.

  • Dirty-state guard

    Write tools refuse to overwrite uncommitted work unless the caller explicitly opts in.

  • Atomic writes

    Edits are validated before application and written with per-path locks and atomic replacement.

  • LSP notification

    After a write, Plumb notifies the language server through watched-file changes so diagnostics stay live.
