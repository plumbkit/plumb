Why plumb

LLMs don't need your files. They need your answers.

Pasting raw source into context is wasteful and imprecise. Plumb gives Claude structured semantic queries instead.

⚡

Shared daemon

One gopls process per workspace, shared across all your conversations. Gopls stays warm between sessions — no re-indexing on every new chat.

🔍

27 tools, multi-language

Symbols, definitions, references, call/type hierarchies, diagnostics, file search, semantic edits, project memories — across Go (gopls) and Python (pyright). Cross-project queries work in a single MCP session.

📍

Workspace-aware

Drop a .plumb directory in your project root. Plumb detects it automatically, even when nested modules or testdata are present.

✏️

Semantic edits

LSP-driven refactoring tools — rename_symbol, replace_symbol_body, insert_before/after_symbol, safe_delete_symbol. Scope-aware, dry-run by default. Git subcommands stay allowlisted to safe, read-only operations.

MCP tools

Twenty-seven tools. Each returns exactly what Claude needs.

Designed to minimise round-trips and eliminate redundant file reads — the source line is already in the response.

find_symbol LSP

Search for a symbol by name across the entire workspace. Returns kind, file path, and line number without reading any file.

get_definition LSP

Jump to the definition of a symbol at a given position. Returns the source with surrounding context lines included.

explain_symbol LSP

Hover documentation — type signature, godoc comment, and declared package path — for a symbol at a cursor position.

list_symbols LSP

Complete symbol hierarchy for a file — structs, methods, fields, line ranges, and nesting depth. No blind searches needed.

find_references LSP

All usages of a symbol, grouped by file, with the source line already included so Claude never needs a second file read.

diagnostics LSP

Live errors, warnings, and hints from the language server, pushed as gopls analyses the workspace. Grouped by file and severity.

list_files filesystem

Walk a directory tree with glob filtering, depth control, and hidden-file toggle. Automatically excludes vendor, node_modules, .git.

git VCS

Read-only git subcommands — diff, log, blame, status, show, stash. Destructive commands are blocked with a clear error.

file_diff filesystem

Unified diff between two arbitrary files with configurable context lines and optional whitespace-ignore mode.

workspace_symbols LSP

Fuzzy-search symbols across the entire workspace by name. Faster than grepping — the language server indexes every declaration.

call_hierarchy LSP

Map who calls a function (incoming) or what it calls (outgoing). Full call chains without reading a single file.

type_hierarchy LSP

Navigate supertypes and subtypes for any interface or struct. Shows the full inheritance graph from one call.

search_in_files filesystem

Ripgrep-style content search — smart-case, regex, glob filter, context lines. Gitignore-aware. No external binary required.

find_files filesystem

fd-style file finder — glob or regex patterns, extension filter, type filter (file/dir), max depth. Gitignore-aware.

version info

Returns plumb's version, Go runtime, and OS/arch. Useful for bug reports and confirming which binary the daemon is running.

Quick start

Up and running in four steps.

01

Install plumb

terminal
$ go install github.com/golimpio/plumb/cmd/plumb@latest

Requires Go 1.26+. gopls must be on your $PATH (go install golang.org/x/tools/gopls@latest).

02

Wire up Claude Desktop

terminal
$ plumb setup claude-desktop
Registered plumb in ~/Library/Application Support/Claude/claude_desktop_config.json
Restart Claude Desktop to apply the change.

This writes the plumb serve entry into Claude Desktop's MCP server config. Restart Claude Desktop after running it.

03

Initialise your workspace

terminal
$ cd /path/to/your/project
$ plumb init
Initialised .plumb at /path/to/your/project/.plumb
Edit .plumb/context.md to describe your project.

The .plumb directory marks your project root and takes priority over nested go.mod files (e.g. in testdata/). Optionally commit it; edit context.md to give Claude project-specific context.

04

Monitor and manage

terminal
$ plumb status   # live session dashboard (TUI)
$ plumb stop     # gracefully stop the background daemon
$ plumb version  # print version and Go runtime info

The daemon starts automatically on first use and persists between Claude sessions — gopls stays warm. Use plumb stop to shut it down cleanly.

Architecture

One daemon. One gopls. Many conversations.

A shared daemon means the language server is always warm — no re-indexing when you open a new chat.

Claude Desktop
MCP client — spawns one plumb serve per conversation
stdio (MCP JSON-RPC) ↓
plumb serve
thin proxy · dials Unix socket · forwards bytes
~/Library/Caches/plumb/plumb.sock ↓
plumb daemon
one long-lived process · workspace pool · one gopls per root
gopls · ~/projects/foo
gopls · ~/projects/bar
Workspace detection — plumb finds the project root by walking up from the file URI: first looking for a .plumb/ directory (run plumb init at your project root), then falling back to the nearest go.mod. The .plumb marker takes priority, preventing nested modules in testdata/ or vendor/ from being selected as the root.
Monitoring & statistics

Know exactly what Claude is doing.

Plumb records every tool call to a local SQLite database. See which tools are used, how fast they are, and which client connected — all from the terminal.

📊

Per-tool statistics

Call count, average latency, P95, total I/O, and error rate — broken down by tool and workspace.

🖥️

Client detection

Plumb identifies which app connected — Claude Desktop, Claude Code, or any other MCP client — via the MCP initialize handshake.

⚡

Live dashboard

The built-in TUI shows active sessions, tool stats, and recent calls in real time. Refreshes every second.

🗄️

SQLite store

All data is stored locally in ~/.local/share/plumb/stats.db. WAL mode lets the TUI and CLI read concurrently while the daemon writes.

Workspace context

Give Claude your project's memory.

The .plumb/context.md file is loaded automatically on every session. Use it to give Claude a persistent overview of your project — no more re-explaining architecture in every chat.

Keep it concise. A few paragraphs covering purpose, key design decisions, and coding conventions is plenty.

  • Overview — what this project does
  • Architecture — key layers and invariants
  • Conventions — naming, testing patterns, non-obvious rules
  • Known gotchas — footguns that have burned you before
.plumb/context.md
# Project Context

## Overview
plumb is an MCP server that exposes LSP
capabilities to LLMs via a shared daemon
architecture.

## Architecture
Strict layered — lower layers never import
higher ones. One gopls per workspace root,
shared across all conversations.

## Conventions
Australian English in all prose.
log/slog only — never fmt.Println.
Errors must wrap context with %w.

## Known gotchas
gopls not started until first tool call.
Session registers only after workspace
root is resolved.
