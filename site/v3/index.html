01

Interface idea

A future terminal should expose intent, not imitate a monitor.

Plumb treats the codebase as a semantic space. The agent asks for meaning: where a symbol lives, who calls it, what type it has, what changed, whether the workspace is healthy. The response is structured enough to act on.

01

Ask at the level of code meaning.

Symbols, references, definitions, hovers, call hierarchies, type hierarchies, and diagnostics come from the language server.

02

Return only what the agent can use.

Source lines, locations, signatures, and metadata arrive directly in the tool response, avoiding follow-up file reads.

03

Edit through guarded primitives.

Per-path locks, atomic writes, mtime checks, rate limits, LSP notifications, and diagnostics keep autonomous edits bounded.

04

Runtime architecture

A shared daemon keeps the workspace alive.

client Claude Desktop / Claude Code / Codex / Gemini CLI

Each conversation connects through MCP over stdio.

proxy plumb serve

Thin stdio proxy. Dials the local daemon socket and forwards bytes.

core plumb daemon

Singleton process. Hosts per-connection sessions, rate limits, stats, and caches.

workspace workspacePool

One language server per workspace root. gopls or pyright stays warm between conversations.
